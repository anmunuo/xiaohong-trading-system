## 三、数据效率 — 🔴 最严重问题

### 瓶颈 #1：get_stock_realtime() 逐只 subprocess

```python
# data_pipeline.py:594 — 当前实现
for code in stock_codes:
    proc = subprocess.run(['data', 'fetch', 'stock', '--symbol', code, ...], timeout=30)
```

**影响面：** 被 stock_recommender / scout / ammo_risk 广泛调用

| 调用场景 | 标的数 | 最坏耗时 | 优化后 |
|:--|:--|:--|:--|
| 推荐引擎过滤 | 50 候选 | 25 分钟 | <5 秒 |
| 推荐引擎打分 | 9 推荐 | 4.5 分钟 | <1 秒 |
| 侦察兵盘中扫描 | 40 候选 | 20 分钟 | <3 秒 |
| 弹药库风控 | 9 持仓 | 4.5 分钟 | <1 秒 |

**优化方案：** 改为 Sina 批量 HTTP（同 sniperd 模式），或 `data fetch` 批量模式

### 瓶颈 #2：stock_recommender 同批数据三重查询

```
run() 调用链:
  _apply_filters()    → get_stock_realtime([code]) ×50  (第1遍)
  _score_technical()  → get_stock_realtime([code]) ×50  (第2遍)
  _calc_stop_loss()   → get_stock_realtime([code]) ×9   (第3遍)
```

同一批候选股被实时查询 3 遍。应在 `run()` 入口一次拉取全量缓存。

### 瓶颈 #3：历史日线逐只 subprocess

sniperd `_load_historical_data()` 和 ammo_risk `_get_avg_amount_5d()` 各自独立逐只拉取 `data fetch daily`，数据重叠但互不共享。
